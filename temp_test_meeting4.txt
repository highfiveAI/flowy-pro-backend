네, 어서오십시오. 오늘 회의는 우리가 작성한 QA 시나리오 기반으로 기능별 테스트 전략을 수립하고, 실제 QA 문서까지 이어지는 흐름을 점검하겠습니다. QA 시나리오는 어제 공유 드렸는데, 다들 확인하셨나요?

"네, 시트 안의 구조는 명확해서 좋았어요. 그런데 테스트 기준이 기능별로 좀 달라서 우선순위 정리부터 먼저 해봐야 될 것 같아요. 프론트 기준으로 보면 사용자 역할에 따라 접근 가능 여부 확인이나 입력값 검증이 반복되는데, 이걸 일일이 QA가 재현하기엔 좀 어렵더라고요. 반복되는 건 QA 케이스 간소화도 검토해볼 필요가 있을 것 같습니다."

"디자인 측면에서도 버튼 색이나 텍스트값이 QA 대상인지 매번 애매하더라고요. 일관된 기준이 있으면 QA에서 지적하기 애매한 부분도 정리될 것 같아요."

"맞아요. 그래서 저는 QA 항목별로 반드시 체크해야 하는 필수 기준이랑 가이드라인 정도로 구분하는 방식이 좋다고 봅니다. 특히 정책상 필수 조건 같은 건 구분해서 반영하면 좋을 것 같아요."

그러면 QA 문서 양식은 현재 시트 형태로 되어 있는데, 이거 노션 템플릿이나 PDF로도 출력 가능하게 정리하겠습니다. 다인님, 템플릿 쪽 시각 가능할까요?

"네. 레이아웃만 정해주시면 피그마로 정리하고 PDF로 변환도 가능합니다."

네, 알겠습니다. 아 참, 그럼 다음 주 워크숍 시간 다들 어떠세요?

"다음 주 목요일 말씀하시는 거죠? 저 그날 병원 진료가 있어서 좀 애매하기는 한데, 오전은 괜찮을 것 같아요."

"저는 그주 수요일까지 좀 바빠서 목요일 오전은 좀 힘들고요. 금요일은 어떻게 안 될까요?"

"금요일 대부분 안 될 것 같은데... 저희 근데 워크숍이라고 해봤자 결국 다 같이 노션 뭐 정리하는 거 아니에요? 장소만 바뀐 정기회의 할 것 같은데..."

"다들 의욕이 넘치시네요. 나중에 구글 폼에 제가 돌려서 응답 받겠습니다."

다시 본론으로 한번 돌아가서요. 그러면은... 우선순위 구분 기준이 사용자 영향도, 그 다음에 API 로직 복합도, 복잡도. 이렇게 두 가지 곱해서 점수화 할까요? 뭐 1에서 3등급 이런 식으로요.

"아, 좋습니다. 사용자 영향도는 고객 피드백 기준으로... 사용자 영향도는 고객 피드백 기준으로 데이터도 줄 수 있는 것 같아요. 지난번에 말했던 오류 유형 통계가 참고가 될 것 같아요."

그럼 각자 Q&A 문서 양식 제작, 실패 시나리오 포함해서 케이스 설계, Q&A 항목 점수와 기준 설계 등 맡아서 해주시면 될 것 같고요. 다음 회의는 다음 주 수요일 오후 3시? 중간 점검 겸 데모도 할까요?

"네, 그러려면 전에 단위 테스트도 다 해야겠네요."

"네, 그럼 저도 피그마 정리하고 노션 양식으로 변환도 해서 한번 공유 드리겠습니다."

네, 알겠습니다. 그럼 오늘 회의 여기서 마무리 하겠습니다. 고생 많으셨습니다.
